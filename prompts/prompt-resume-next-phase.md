## 今回の重要な新事実

これまで主に想定していた流れは、

`NotebookLM音声生成 -> 再アップロード/書き起こし -> 疑似TTS台本化 -> NLMYTGen -> YMM4`

でした。

しかし新たに、NotebookLM に

`「音声解説の元の台本を出力してください」`

と依頼すると、誤字脱字が少なく、話者ラベル付きのクリーンな台本が得られることが分かりました。

想定される新しい主導線は、

`NotebookLMソース投入 -> Audio Overview生成 -> 元台本を出力 -> NLMYTGen -> YMM4`

です。

補足:

- 出力例は `スピーカー1: ...` / `スピーカー2: ...` のようなラベル付き対話です
- コピペ時に句読点や改行が少し崩れることはありえます
- 音声再アップロード経由は、今後は fallback 候補です

## 今回の相談テーマ

台本取得の問題はかなり軽減されそうですが、動画全体のクオリティはまだ別問題として残っています。

今後やりたくなる候補には、たとえば以下があります。

- ゆっくり解説形式への自動変換
- 背景動画演出のデザインと実装
- サムネイル生成
- YMM 側に渡す演出情報や API 相当の中間形式づくり

ただし、これらは LLM 連携を入れた瞬間に、

- 複雑化
- 環境依存化
- 再現性低下
- 責務の越境

を強く招く懸念があります。

この repo は過去に「品質生成の責務が Python 側へ流れ込んで壊れた」経緯があります。今回も同じ失敗を避けたいです。

8. 必要なら docs 更新案まで出す

## 強い制約

以下の原則は崩さないでください。

- `NotebookLM is upstream`
- Python は品質の本体を生成しない
- Path A only (`NotebookLM -> NLMYTGen -> YMM4`)
- direct video generation はやらない
- mandatory な cloud runtime は入れない
- 未来のための汎用抽象化を先回りで入れない
- 他プロジェクトを参照しない
- Web UI / API / YouTube 連携はまだ主対象にしない

LLM を使う案を出すこと自体は構いませんが、必ず以下のどれかに閉じ込めてください。

- 手動運用の補助
- optional な前処理
- repo 外の補助ツール
- 明確に切り離された境界の外側

逆に、以下は避けてください。

- repo の中心ロジックが LLM 必須になる設計
- 1 ステップでサムネ / 演出 / 台本変換 / 動画生成を全部つなぐ巨大統合
- 先に複雑な抽象レイヤーだけ作ること
