# S-6 背景アニメーション 次の一手以降 プラン設計準備

再調査結果を受けて、意思決定を迷走させずに次フェーズへ進むための準備シート。
目的は「提案を増やすこと」ではなく、「実制作の手間削減に直結する最小実行」を選ぶこと。

---

## 1. この準備シートのゴール

- 再調査の結論を、`着手可否` と `着手順` に変換する。
- `NLMYTGen` の境界を守ったまま、S-6 の重作業（背景アニメ）を削減する。
- PoC から実運用までを 1 本の判断線でつなぐ。

---

## 2. 入力チェック（Go/No-Go 前提）

次の 5 条件が埋まっていない場合は、設計に進まない。

- 一次情報の根拠リンクが明示されている
- 候補案ごとに境界適合判定がある（適合/条件付き/不適合）
- 失敗モードが failure class で定義されている
- PoC 合格条件が機械的に判定可能
- 既存運用（CSV -> YMM4 -> apply-production）への接続点が明記されている

---

## 3. 判断ゲート（段階通過方式）

### Gate A: 境界適合

- 質問: 現行境界に違反せず導入できるか？
- 合格条件:
  - Python 画像生成・合成を前提にしない
  - 非公開 API 依存を前提にしない
  - `.ymmp` ゼロ生成を前提にしない
- 不合格時の扱い:
  - 案を破棄するか、境界内に再設計して再提出

### Gate B: 価値検証

- 質問: S-6 の作業時間を実際に減らせるか？
- 合格条件:
  - 対象作業（例: 背景切替、背景アニメ設定、タイミング調整）が特定されている
  - 1 本の制作フローで削減効果を計測できる
- 不合格時の扱い:
  - 価値が曖昧な案は hold

### Gate C: 運用耐性

- 質問: 更新・環境差で壊れにくいか？
- 合格条件:
  - 依存バージョンと破断点が明示
  - 失敗時に fallback（手動運用へ復帰）がある
- 不合格時の扱い:
  - proof-only に留めて本番導入しない

### Gate D: 実装可能性

- 質問: 今のチーム/時間で着手可能か？
- 合格条件:
  - 2週間以内に PoC を閉じられる
  - テスト/検証手順が明示されている
- 不合格時の扱い:
  - 90日ロードマップ後段へ移動

## 5. 次の一手（決定後の実行順）

### Step 1: 台帳反映（必須）

- 採択した案を `FEATURE_REGISTRY.md` に `proposed` で追加
- ユーザー承認後に `approved` へ変更
- 承認前に実装しない

### Step 2: PoC 設計（薄く早く）

- 1 目的 1 PoC に限定する
- 完了条件を「見た目が良い」ではなく「判定可能な指標」で定義する
- 例:
  - 指定した背景アニメパターンが write/readback で一致
  - failure class が期待通りに検出される

### Step 3: 検証運用（既存フロー接続）

- `apply-production` を含む実フローで 1 本検証
- 追加の手作業が増えたら採択を見直す

### Step 4: 本番導入判定

- PoC 成功 = 即本番ではない
- Operator 観点で「体感工数が減ったか」を確認して Go
