# B-15: 改行コーパス収集フォーマット

## 目的

B-14 で bulk overflow は収束した。残課題は個別ケースの改行品質。
heuristic を足し続けるより、「改行すべき / すべきでない例」を集めて傾向化する方が強い。

このドキュメントはコーパスの記録フォーマットを定義する。

## 収集対象

YMM4 取込後に人間が「直した」または「直したくなった」改行箇所。
以下の 4 分類で記録する。

| 分類 | 意味 | 例 |
|------|------|-----|
| bad-split | 不自然な位置で切れている | 単語途中、助詞が次行に孤立 |
| bad-merge | 別文が1行に結合されている | 句点後に次文が続く |
| missing-split | 切るべきなのに切れていない | 長すぎる1行 |
| good | 良い改行の例（保持すべき） | 読点後の自然な分割 |

## 記録フォーマット (TSV)

ファイル: `samples/<動画名>_linebreak_corpus.tsv`

```
line	category	before	after	note
15	bad-split	「アルゴリズムによる最適化」\nと聞くと、	「アルゴリズムによる最適化」と聞くと、\n効率的でクリーンな	括弧+引用直後で分割、「と」が孤立
```

| 列 | 説明 |
|----|------|
| line | CSV の行番号 |
| category | bad-split / bad-merge / missing-split / good |
| before | 分割前のテキスト（\n で改行位置を示す） |
| after | 望ましい形（人間が直した結果、または理想形） |
| note | 問題の原因メモ（ルール化の手がかり） |

## 傾向化の基準

- 同類の問題が 3 件以上 → ルール候補として検討
- 1-2 件のみ → 許容範囲として受容、コーパスに記録のみ
- 相反する例（同じパターンで good と bad が混在）→ 文脈依存として記録

## コーパスの利用

1. 傾向が見えたら `assemble_csv.py` の定数または分割ロジックに反映
2. テストケースとして `test_split_long_utterances.py` に追加
3. 相反例が多い場合は heuristic 追加を見送り、「現状で十分」と判断する材料にする
