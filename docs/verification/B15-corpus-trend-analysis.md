## 傾向パターン (3件以上で「ルール候補」)

### P1: 閉じ括弧直後での分割 (5件) -- ルール候補

該当: L15, L31, L52, L73, L85

```
「最適化」\nと聞くと
「限界」\nという変数
「摩擦」\nを一切
「猛暑」\nという
（DSP）\n」プログラム
```

**共通原因**: `_AGGRESSIVE_BREAK_AFTER` が `」` `）` をペナルティ1で候補化しているが、
閉じ括弧の直後は助詞 (と/という/を) や開き括弧ペアの残りが来るため、
分割すると意味のまとまりが崩れる。

**対策案**: 閉じ括弧直後に助詞 (と/を/が/は/に/で/も/の/という/として) が続く場合、
その助詞の後まで分割を延長する。

### P2: 左側が極端に短い (3件) -- ルール候補

該当: L41, L73, L96

```
例えば、\n (4文字)
「猛暑」\n (4文字)
それに対し、\n (7文字)
```

**共通原因**: `min_chunk = max(6, max_length // 4)` で最小チャンク幅を制限しているが、
balance_subtitle_lines の `_balance_two_lines` には同等のガードがない可能性。
または split_long_utterances 側の min_chunk が小さすぎる。

**対策案**: min_chunk を引き上げるか、左側の最小幅ガードを強化する。

### P3: 単語途中での切断 (1件) -- 保留

該当: L45 (`見せつけられなが\nら`)

**原因**: chars_per_line による強制切断。文字数制限で候補が見つからなかった場合のフォールバック。
1件のみなので保留。

## good パターンの特徴

4件とも「読点 `、` の直後で分割」かつ「左右のバランスが概ね均等」。
現在のアルゴリズムは読点分割時に最も良い結果を出している。

## 次のアクション

- P1 (閉じ括弧+助詞) と P2 (左側短すぎ) は各3件以上 → ルール改善の価値あり
- ただし、実装前にユーザーの手動検証を挟む:
  1. このコーパスの分類と対策案が妥当か
  2. 「直した」結果 (after列) が実際に望ましいか
  3. P1/P2 のルール改善を承認するか
