# B-11 運用: 手動確認が必要なタイミング

[workflow-proof-template.md](workflow-proof-template.md) に沿った記録を取る際、**機械（CLI/GUI）が保証しない**箇所と、**人が必ず見るべき順序**を整理する。

## 1. 取込前（`build-csv`）


| タイミング                  | 誰が    | 内容                                                                   |
| ---------------------- | ----- | -------------------------------------------------------------------- |
| `--stats` / Dry Run 直後 | オペレータ | 話者数・行数・overflow 警告が期待どおりか。ここは **YMM4 を開かずに** 判断できる。                  |
| 初めて使う台本形式              | オペレータ | `--unlabeled` や speaker map の要否。必要なら `inspect` / `generate-map` で確認。 |


**手動で不要なこと**: ピクセル一致の字幕プレビュー（F-03 は rejected。近似幅のみの簡易表示は別 FEATURE）。

## 2. YMM4 取込〜S-5


| タイミング      | 誰が    | 内容                                                            |
| ---------- | ----- | ------------------------------------------------------------- |
| 台本読込直後     | オペレータ | キャラクター割当・話者名の取り違えがないか。                                        |
| 読み上げ・字幕の通し | オペレータ | **読み**と**字幕の見た目**（はみ出し・不自然改行・辞書）。ここが B-11 の **取込後 4 区分**の実測元。 |
| 辞書登録       | オペレータ | 固有名詞の読み。機械では代替しない。                                            |


**記録のコツ**: 修正した字幕行や CSV 行番号が分かればテンプレの「代表例」にメモしておくと、次のコーパス化が速い。

## 3. S-6（演出 IR・apply-production）


| タイミング                                                  | 誰が             | 内容                                                                                                                                   |
| ------------------------------------------------------ | -------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `validate-ir` / Apply 前                                | オペレータ（補助: CLI） | palette / IR / row-range の機械ゲート。GUI の要約 JSON は **件数と先頭プレビュー**に留まるため、エラー全文はログを見る。                                                     |
| patch 後の ymmp                                          | オペレータ          | 表情・背景の**見え方**・テンポ。創作判断。                                                                                                              |
| `TIMELINE_ROUTE_MISS` / `SE_WRITE_ROUTE_UNSUPPORTED` 等 | オペレータ＋設計       | 多くは **その ymmp サンプルに経路が無い**という mechanical 区分。別テンプレで再測定するか、期待 route を更新するかの判断（[OPERATOR_WORKFLOW.md](OPERATOR_WORKFLOW.md) G-12/G-13）。 |


## 4. 品質診断（H-03 / H-04）


| タイミング     | 誰が    | 内容                                                                                      |
| --------- | ----- | --------------------------------------------------------------------------------------- |
| Brief の中身 | オペレータ | H-01 brief は人が埋める（または LLM 補助でも**最終確定は人**）。空テンプレは `emit-packaging-brief-template` で出力可能。 |
| カテゴリスコア入力 | オペレータ | GUI の 0〜3 は**主観プロキシ**。スコア結果は診断用で、公開判断の自動化ではない。                                          |


## 5. 外部 LLM 用パケット（cue / diagram）


| タイミング       | 誰が    | 内容                                                                                                                                 |
| ----------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------- |
| パケット生成後     | オペレータ | 外部ツールへ貼付。**生成物は text-only**。主台本のゼロ生成や ymmp 直接編集は境界外（[ADR/0004-llm-text-assist-boundary.md](ADR/0004-llm-text-assist-boundary.md)）。 |
| LLM 出力の取り込み | オペレータ | IR / 台本への反映は人がレビューしてから。                                                                                                            |


## 6. 次のアクション（推奨）

1. **実案件 1 本**で [workflow-proof-template.md](workflow-proof-template.md) をコピーし、取込後表を**実数**で埋める（サンプル例は [verification/B11-workflow-proof-sample-example-dialogue.md](verification/B11-workflow-proof-sample-example-dialogue.md)）。
2. 残修正が「例外」に寄ったかを [OPERATOR_WORKFLOW.md](OPERATOR_WORKFLOW.md) B-11 の判断ゲートに 1 行追記。
3. 改行痛点が残るなら `tests/data/linebreak_forbidden_corpus.json` 等へ文例を足し、ルール改善の材料にする。
