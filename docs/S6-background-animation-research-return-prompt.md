# S-6 背景アニメーション再調査 依頼プロンプト

既存調査の価値は認めつつ、`NLMYTGen` の現行境界に合わせて再調査を差し戻すための依頼文。
そのまま外部調査担当（LLM/人手）へ渡して使う。

---

## コピペ用プロンプト本体

```
あなたは、YMM4 運用と NLMYTGen のプロジェクト制約を理解した技術調査担当です。
以下の条件を厳守し、背景アニメーション自動化の再調査を実施してください。

【目的】
- S-6（背景・演出設定）で最も重い「背景アニメーション」の手作業を減らす。
- ただし、現行のプロジェクト境界（Python責務・YMM4責務）を破らない。
- 結果は「次の実装判断に直結する意思決定資料」として提出する。

【前回調査の評価（前提）】
- 良い点:
  - 「外部で重い計算、YMM4で最終合成」の分離思想
  - YMM4拡張（プラグイン/エフェクト/素材規約）を一次情報で押さえている点
- 問題点:
  - 現行境界では Python 画像合成・画像生成は採用不可
  - 一部提案が NLMYTGen の責務外（または未承認機能）を含む
- このため、再調査では「実装可能性」を最優先し、境界違反案を除外した構成にすること

【厳守する境界（必須）】
1) Python 側で画像生成・画像合成を前提にしない
2) .ymmp のゼロから生成を前提にしない
3) YMM4 非公開 API 依存を推奨しない
4) 公開情報・公開 API・公開仕様を根拠にする
5) 既存運用（CSV -> YMM4 読込 -> apply-production）に接続できる提案に限定する

【再調査で答えるべき質問】
Q1. 背景アニメーションを YMM4 側で実現する際、公開情報ベースで安定運用できるルートは何か？
Q2. そのルートは NLMYTGen の既存機能（G-12〜G-18、timeline profile、overlay/se）とどう接続できるか？
Q3. 導入コスト（実装・運用・破断リスク）に対して、どの順番で導入すると最短で制作時間を削減できるか？
Q4. 「今やるべき最小スコープ」と「保留すべき拡張スコープ」をどう分けるべきか？

【提出フォーマット（必須）】
以下の 6 章構成で提出すること。章タイトルは固定。

1. 事実確定（一次情報）
   - 使った一次情報の URL / 版 / 最終確認日
   - 何が「確定」、何が「未確認」かを分離

2. 境界適合性判定
   - 各候補案を `適合 / 条件付き適合 / 不適合` で分類
   - 不適合理由は境界ルール番号に紐づける

3. 実装候補（最大 3 案）
   - 案A/B/C のみ（3案を超えない）
   - それぞれに:
     - 期待効果（S-6 のどの手作業をどれだけ減らすか）
     - 依存（YMM4プラグイン、運用ルール、テンプレ資産など）
     - 破断リスク（YMM4更新、非公開依存、保守負荷）
     - PoC 完了条件（機械的に合否判定できる条件）

4. 推奨順序（90日ロードマップ）
   - 0〜30日、31〜60日、61〜90日の3区分で記述
   - 各区分で「やること」「やらないこと」「判断ゲート」を明記

5. 失敗モードとガードレール
   - 想定失敗を failure class として列挙
   - 各 failure class に検出方法（dry-run/readback/目視）を対応づける

6. 最終提案（1ページ要約）
   - 推奨案を1つに絞る
   - 理由を「価値」「安全性」「保守性」「実装速度」の4軸で記述
   - 「今回見送り」にした案の見送り理由も記載

【禁止事項】
- 画像処理ライブラリ実装（MediaPipe/OpenCV/Pillow 等）を主経路として提案しない
- 未確認情報を確定事実として書かない
- 提案数を増やしすぎて比較不能にしない
- 「とりあえず全部試す」形式にしない

【評価基準（この順で重視）】
1. プロジェクト境界への適合
2. S-6 作業時間の実削減見込み
3. 破断しにくさ（YMM4更新耐性）
4. 実装・検証の速さ
5. 将来拡張性
```

---

## 使い方メモ

- まずこのプロンプトで再調査を実施する。
- 回収した結果は、`docs/verification/` 配下に proof として保管する。
- 採択候補が出たら、`FEATURE_REGISTRY.md` に `proposed` で登録してから次フェーズへ進む。
